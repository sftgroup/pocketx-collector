## 三、用户操作流程

### 3.1 首次使用托管模式
```
打开 PocketX → 选择「创建托管钱包」→ 输入邮箱 → 收到邮件验证码 → 输入验证码
→ 设置支付密码（6 位） → 钱包创建成功 → 显示托管钱包地址
→ 首页展示 0 余额 + 「充值」引导按钮
```

### 3.2 托管模式转账
```
点击 Send → 输入收款地址 → 输入金额 → 系统检查风控规则
→ [通过] 显示「Gas 由 PocketX 代付」标签 → 输入支付密码 → 交易广播 → 显示成功
→ [不通过] 显示拦截原因（超限额/黑名单地址/新用户限制）
```

### 3.3 外部入金
```
外部地址向用户托管钱包转账 → 链上确认 → CWallet 监听到入金
→ Webhook 推送 PocketX 前端 → 前端弹通知「收到 xx USDC」→ 余额自动刷新
```

### 3.4 双模式切换
```
设置页 → 钱包管理 → 切换模式（非托管 ↔ 托管）
→ 非托管模式：连接 MetaMask/WalletConnect
→ 托管模式：输入支付密码验证身份
→ 切换后首页数据刷新为新模式数据
```

## 四、B2B 两种模式

### 模式一：SaaS 钱包即服务（Wallet-as-a-Service）

**目标客户**：交易所、OTC 平台、支付平台、GameFi 项目方等需要托管钱包基础设施的中心化应用

**价值主张**：客户无需自建链上节点、私钥管理、扫链归集系统——调用 PocketX API 即可拥有完整的托管钱包能力

#### 接入流程
```
客户注册企业账号 → 创建租户 → 获取 API Key + Secret
→ 配置 Webhook 回调 URL → 配置归集地址 + 归集阈值
→ 接入 API → 上线运营
```

#### API 服务清单

| 接口 | 方法 | 说明 |
|------|------|------|
| 分配地址 | POST /api/v1/saas/address | 为客户用户创建托管地址，传入 user_id，返回链上地址 |
| 查询地址 | GET /api/v1/saas/address/{user_id} | 查询 user_id 对应地址及其余额 |
| 发起提币 | POST /api/v1/saas/withdraw | 客户用户提币请求，传入 user_id、收款地址、金额、币种 |
| 审核提币 | POST /api/v1/saas/withdraw/{id}/approve | 客户审核通过提币 |
| 拒绝提币 | POST /api/v1/saas/withdraw/{id}/reject | 客户审核拒绝提币 |
| 交易记录 | GET /api/v1/saas/transactions | 查租户所有交易流水 |
| 余额概览 | GET /api/v1/saas/balances | 查租户总余额、各币种分布、待归集金额 |
| Webhook 回调 | — | 入金通知、出金结果、提币请求（审核模式） |

#### SaaS 业务逻辑

| L-ID | 业务规则 |
|------|---------|
| L-018 | 租户隔离：每个企业客户独立租户空间，API Key 隔离，地址池和数据不可互相访问 |
| L-019 | 地址分配：客户调用 API 传入 user_id → 生成唯一链上地址，同一 user_id 始终返回同一地址 |
| L-020 | 自动归集：用户地址入金 ≥ 归集阈值 → 自动归集到客户配置的归集地址 → Webhook 通知 |
| L-021 | 提币审核：用户提币 → Webhook 推送客户 → 客户审核（通过/拒绝）→ 通过则自动打币 |
| L-022 | 提币免审：客户可选择关闭审核模式，提币请求直接自动打币，仅做风控检查 |

#### 用户操作流程（SaaS 模式一）

##### 3.5 客户接入流程
```
客户注册 → 创建租户 → 保存 API Key → 配置 Webhook URL
→ 设置归集地址 → 设置归集阈值（如 100 USDC 自动归集）
→ 选择审核模式（审核/免审）→ 接入完成
```

##### 3.6 客户用户充币流程
```
客户调用 POST /api/v1/saas/address → 获取托管地址
→ 客户向其用户展示该地址（可嵌在充值页面）
→ 用户向该地址转账 → 链上确认
→ PocketX 扫链检测到入金 → Webhook 推送客户（含 txHash、金额、user_id）
→ 入金金额达到归集阈值 → 自动归集到客户归集地址
```

##### 3.7 客户用户提币流程（审核模式）
```
客户用户在前端提交提币 → 客户调用 POST /api/v1/saas/withdraw
→ PocketX 创建提币请求 → Webhook 推送客户审核端
→ 客户审核通过 → PocketX 签名广播交易 → 结果回调客户
→ 客户审核拒绝 → 回调客户拒绝原因，资金退回用户余额
```

##### 3.8 客户用户提币流程（免审模式）
```
客户用户在前端提交提币 → 客户调用 POST /api/v1/saas/withdraw
→ PocketX 检查风控规则（单笔限额、日累计、黑名单）
→ 通过检查 → 直接签名广播交易 → 结果回调客户
→ 不通过 → 拒绝并返回原因
```

### 模式二：机构链上多签（On-Chain Multi-Sig Vault）

**目标客户**：对冲基金、DAO 金库、家族办公室、企业财务部门

**价值主张**：在 PocketX 平台直接创建 Gnosis Safe 多签钱包，M-of-N 管理，链上透明不可篡改，单一平台管理个人钱包 + 多签金库

> 决策：链上多签 — 基于 Gnosis Safe 智能合约钱包，M-of-N 签名，链上透明不可篡改。

#### 功能清单

| F-ID | 功能 | 说明 |
|------|------|------|
| **F-027** | **创建机构多签钱包** | 选择 M-of-N 阈值、添加 Owner 地址、部署 Safe 合约 |
| **F-028** | **多签交易发起** | 任意 Owner 发起交易 → 生成待审批交易 |
| **F-029** | **多签审批** | 其他 Owner 在 PocketX 查看待审批交易 → 签名确认 |
| **F-030** | **交易执行** | 达到阈值后任何人可执行 → Safe 合约验证签名 → 广播链上交易 |
| **F-031** | **多签钱包管理** | 查看 Owners 列表、阈值、pending 交易、历史记录 |
| **F-032** | **Owner 管理** | 添加/移除 Owner、修改阈值（需达到当前阈值审批） |

#### 多签业务逻辑

| L-ID | 业务规则 |
|------|---------|
| L-013 | Safe 部署：机构按需创建 Safe 钱包，支持 1-50 个 Owner，阈值 1-N |
| L-014 | 交易生命周期：发起 → 收集签名 → 达到阈值 → 任何人可执行 → 链上确认 |
| L-015 | Owner 变更：添加/移除 Owner 或改阈值本身也是一笔多签交易，需达到当前阈值 |
| L-016 | 多链支持：Safe 已部署在 Ethereum/Polygon/Arbitrum/Optimism/BSC/Base |
| L-017 | Gas 支付：执行交易时由执行者支付 Gas（非代付模式）。未来可选 PocketX Gas Pool 代付 |

#### 用户操作流程（SaaS 模式二）

##### 3.9 创建机构多签钱包
```
设置页 → 钱包管理 → 「创建机构多签钱包」
→ 输入钱包名称（如「XX 基金·金库」）
→ 添加 Owner 地址（至少 2 个）
→ 设置签名阈值（如 3/5）
→ 确认创建 → PocketX 调用 Safe SDK 部署 Safe 合约
→ 返回 Safe 合约地址 → 加入钱包列表
```

##### 3.10 多签交易全流程
```
Owner A 发起交易：
  钱包中选择 Safe 钱包 → Send → 输入收款地址+金额
  → 显示「需 3/5 签名（当前 0/5）」→ 确认签名

Owner B/C 审批：
  收到推送通知「多签待审批」→ 打开 PocketX
  → 切换到对应 Safe 钱包 → 待审批列表
  → 查看交易详情 → 点击「确认签名」→ 当前 2/5

Owner D 确认+执行：
  签名 → 当前 3/5 → 达到阈值！
  → 按钮变为「执行交易」→ 点击 → 支付 Gas → 广播上链
  → 交易完成，所有 Owner 收到通知
```

## 九、B2B 两种模式对比

| 维度 | 模式一：SaaS WaaS | 模式二：机构多签 |
|------|-------------------|-------------------|
| **目标客户** | 交易所、OTC 平台、支付平台、GameFi | 对冲基金、DAO 金库、家族办公室、企业财务 |
| **核心需求** | 为用户托管资产，需要充提基础设施 | 团队共同管理大额资产，需要多人审批 |
| **钱包模型** | PocketX 托管，用户不持有私钥 | Gnosis Safe 链上合约，M-of-N 签名 |
| **接入方式** | REST API + Webhook | PocketX 前端界面 |
| **私钥管理** | PocketX HSM 加密托管 | 各 Owner 用自己的 EOA 钱包签名 |
| **审核机制** | 客户审核提币（可选免审） | 多签阈值（链上强制） |
| **归集功能** | ✅ 自动归集到客户地址 | ❌ 无需归集 |
| **Gas 处理** | PocketX Gas Pool 代付 | 执行者自行支付 Gas |
| **并发用户** | 多用户（客户平台下的终端用户） | 多 Owner（同一钱包的签名者） |
| **关键指标** | 地址数、日入金量、归集效率 | 签名数、执行成功率、审批耗时 |
